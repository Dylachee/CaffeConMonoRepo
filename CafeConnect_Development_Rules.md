# CafeConnect — Правила разработки (Development Rules)

> Этот файл — короткий свод правил, по которым строится хорошее приложение. Принципы взяты
> из проверенной практики (ECC — система правил для ИИ-агентов) и **адаптированы под Flutter / Dart
> и под этот проект**. Давай этот файл любому ИИ-агенту вместе с Vision и Implementation Prompt.
> Здесь нет шагов «что делать» — здесь правила «как делать, чтобы было хорошо».
>
> Язык кода: **Dart**. Платформа: **Flutter**. Эти правила всегда в силе.

---

## 0. Самое главное правило — дисциплина сохранения работы

Это правило важнее всех остальных, потому что без него теряется вся работа.

- **Нет в репозитории на GitHub — значит не существует.** ИИ-агент работает **только локально**: создаёт, меняет, удаляет файлы. Он **никогда** не делает `git commit`, `git push`, не создаёт PR, теги, релизы, не трогает remote.
- Закончив, агент отдаёт **список изменённых файлов** + одну строку на каждый, что изменено, + что проверить руками. **Коммитишь и пушишь ты сам**, после проверки.
- Перед началом крупной работы — сохрани текущее состояние сам (твой commit + push), чтобы было куда откатиться.

*Почему: работа ИИ-агентов уже не раз «испарялась» при обрыве контекста. Это правило чинит главную боль надёжнее любого инструмента.*

---

## 1. Сначала исследуй, потом меняй (Research-first)

- **Прочитай весь связанный код, прежде чем что-то менять.** Понять → потом править. Никаких изменений «вслепую».
- Не переписывай рабочий код без причины. Не меняй архитектуру и не переименовывай без веской причины. **Не удаляй функциональность.**
- Если задача неоднозначна — **оставь комментарий с вопросом в коде**, а не угадывай. Не выдумывай требований, которых нет.
- Можешь исправить локально — исправляй локально. Не тащи новые зависимости без необходимости.

*Почему: большинство поломок рождается из изменений в коде, который агент не понял целиком.*

---

## 2. Неизменяемость (Immutability) — КРИТИЧНО

- **Всегда создавай новый объект, никогда не мутируй существующий.** Возвращай копию с применёнными изменениями.
- В Dart: используй `final` по умолчанию, `copyWith(...)` для изменений моделей, неизменяемые поля.

```dart
// ПЛОХО — мутация
table.guestCount = 4;

// ХОРОШО — новый объект через copyWith
final updated = table.copyWith(guestCount: 4);
```

- Модели данных (Table, Order, OrderItem, …) — неизменяемые, с `copyWith` и понятными полями.

*Почему: неизменяемые данные убирают скрытые side-effect'ы, упрощают отладку и делают состояние предсказуемым — критично для realtime-приложения, где один и тот же стол виден многим.*

---

## 3. Организация файлов — много маленьких, а не несколько огромных

- **Много маленьких файлов лучше нескольких больших.** Типичный размер файла **200–400 строк**, **максимум ~800**.
- **Группируй по фиче/домену, а не по типу.** То есть `tables/`, `orders/`, `menu/` — а не «все модели в одну папку, все экраны в другую».
- Высокая связность внутри модуля, слабая связь между модулями (high cohesion, low coupling).

*Почему: твой `main.dart` на ~2817 строк — это ровно та боль, которую это правило лечит. В большом файле тонет и человек, и ИИ.*

> Примечание для этого проекта: разбивать монолит на модули — **отдельная большая задача**. Делать её
> только осознанно и одним заходом, не посреди багфикса. Пока не разбит — фиксь баги внутри него.

---

## 4. Понятные имена

- Имена переменных, функций, классов — **описательные**. Видно назначение без комментария.

```dart
// ХОРОШО
final marketSearchQuery = 'election';
final isUserAuthenticated = true;
final totalRevenue = 1000;

// ПЛОХО
final q = 'election';
final flag = true;
final x = 1000;
```

- Именованные константы для значимых порогов, задержек, лимитов — не «магические числа» в коде.

```dart
// ХОРОШО
const lateOrderThreshold = Duration(minutes: 20);
if (elapsed > lateOrderThreshold) ...

// ПЛОХО
if (elapsed.inMinutes > 20) ...
```

---

## 5. Читаемость кода

- **Ранний возврат (early return)** вместо вложенных условий, когда логика начинает «лесенкой» нарастать.

```dart
// ХОРОШО
if (order == null) return;
if (order.items.isEmpty) return;
process(order);

// ПЛОХО — вложенность
if (order != null) {
  if (order.items.isNotEmpty) {
    process(order);
  }
}
```

- Один виджет/функция — одна ответственность. Длинный `build()` — повод вынести под-виджеты.

---

## 6. Обработка ошибок — на каждом уровне, никогда молча

- **Обрабатывай ошибки на каждом уровне.** В UI — понятное человеку сообщение. В сервисах/логике — детальный контекст в лог.
- **Никогда не глотай ошибку молча** (пустой `catch {}` — запрещён).
- Для оптимистичных обновлений (правило проекта): применили локально → при ошибке откат + тост пользователю. Ошибка не должна «исчезать».

```dart
try {
  final result = await api.submitOrder(order);
  // примирить состояние с ответом
} catch (e, st) {
  // откатить оптимистичное изменение
  revertOrder(order);
  // понятный тост пользователю
  showToast('Не удалось отправить заказ. Повторите.');
  // детальный лог для отладки
  log('submitOrder failed', error: e, stackTrace: st);
}
```

---

## 7. Проверяй вводимые данные на границах

- **Валидируй любой ввод на границе системы.** Не доверяй внешним данным (ввод официанта, ответ сервера, сигнал от гостевого web).
- Fail fast — падай рано и с понятным сообщением, а не тихо продолжай с битыми данными.
- Пример из проекта: число гостей ≥ 1; превышение `seats` — мягкое предупреждение, а не краш; входящий `AttentionSignal` с неизвестным `type` — не должен ронять экран.

---

## 8. Тестирование — хотя бы критические потоки

- Полное TDD и «80% покрытия» из ECC — это для зрелого продукта. Для тебя сейчас разумный минимум:
- **Покрой тестами критические потоки**, которые ломаются молча: авто-сплит заказа (еда→кухня, напитки→бар), пометка «отдано» обновляет счётчик, сигнал гостя перекрашивает нужный стол, состояние сохраняется и читается из Hive.
- Когда чинишь баг — **сначала тест, который его воспроизводит**, потом фикс. Так баг не вернётся.
- Падает тест — чини **реализацию**, а не тест (тест меняешь, только если он сам неверный).
- В конце: `flutter analyze` без ошибок, тесты зелёные. Сломанный/устаревший тест замени на реальный.

---

## 9. Производительность и контекст (для работы с ИИ)

- **Не усложняй.** Относись к конфигурации как к тонкой настройке, а не как к архитектуре. Не строй сложную систему там, где хватит простого.
- **Контекст ИИ — дорогой ресурс.** Давай агенту ровно то, что нужно для задачи (этот файл + Vision + Prompt + нужные файлы), а не «всё подряд».
- Отключай/не подключай то, чем не пользуешься (лишние MCP, плагины, зависимости).

*Почему: это прямое противоядие твоей повторяющейся ловушке «нахватать всё». Меньше — надёжнее.*

---

## 10. Безопасность — базовый минимум

- **Никаких секретов в коде.** Ключи API, токены, пароли — не хардкодить (сейчас в проекте демо-логины вшиты прямо в код — это заглушка, не образец; реальная авторизация так не делается).
- Не доверяй внешним данным (см. §7). Проверяй то, что приходит снаружи.
- Сообщения об ошибках пользователю — без технических подробностей и без утечки внутренних данных.

---

## 11. Единый стиль и минимум побочных изменений

- Держи **один стиль** именования, форматирования и структуры — как уже принято в проекте.
- **Минимизируй побочные изменения.** Не трогай то, что не относится к задаче. Не меняй окружение, конфиги и зависимости без явной необходимости.
- Новые виджеты — строго в существующей дизайн-системе (токены цвета, шрифты Inter/JetBrains Mono).
- **Никаких временных хаков. Никаких оставленных `TODO`.** Существенные архитектурные изменения — задокументируй.

---

## 12. Рабочий цикл одной задачи (коротко)

Когда ИИ берёт задачу — порядок такой:

1. **Изучи** связанный код целиком (§1). Определи зависимости и риски.
2. **Меняй** по этим правилам: неизменяемость, маленькие файлы, понятные имена, обработка ошибок.
3. **Проверь** — ничего не сломалось: `flutter analyze` чисто, тесты зелёные, критический поток работает.
4. **Отчитайся** — список изменённых файлов + что проверить руками. **Не коммить, не пушь** (§0).

---

### Свод в одном абзаце
Работай локально и никогда не трогай git — сохраняет и пушит человек. Сначала пойми код целиком, потом меняй; не ломай рабочее, не удаляй функциональность. Данные — неизменяемые (`copyWith`, `final`). Файлы маленькие (200–400 строк), сгруппированы по фиче. Имена описательные, числа — именованные константы, ранний возврат вместо вложенности. Ошибки лови на каждом уровне и показывай человеку понятно, никогда не молчи. Внешние данные проверяй на границе. Тестами закрой критические потоки, баг чини тестом-репродьюсером. Не усложняй, не тащи лишнего, не хардкодь секреты, держи единый стиль и минимум побочных правок. В конце — отчёт о файлах, а коммит за тобой.
